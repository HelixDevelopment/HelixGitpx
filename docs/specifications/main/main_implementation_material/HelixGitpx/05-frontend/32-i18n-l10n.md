# 32 — Internationalisation & Localisation (i18n / l10n)

> **Document purpose**: Define how HelixGitpx supports **multiple languages, regions, and cultural conventions** across web, mobile, desktop, email, and APIs. Every user-visible string is translatable; every user-visible format (dates, numbers, currency) respects locale.

## 2. Principles

1. **Everything translatable.** No hardcoded strings in user-visible code paths — always go through `t()` or `i18n` API.
2. **ICU MessageFormat.** Plurals, gender, selects, numbers, dates natively.
3. **Locale-aware formatting.** Never `toString()`; use `Intl.*` or platform equivalent.
4. **RTL-safe layouts.** All components tested in LTR and RTL.
5. **Source of truth is English.** Translations review-gated; missing translations fall back to English.
6. **Continuous translation.** CI pushes new keys to the translation platform; weekly PRs merge completed translations.

---

## 3. Technical Stack

### 3.1 Web (Angular)

- **ngx-translate** + **ICU MessageFormat parser** (or Angular's built-in `$localize`).
- Message files: `libs/i18n/<locale>/messages.json` — flat key-value with ICU.
- Locale negotiation: user setting → `Accept-Language` → browser default → English.
- Bundle: per-locale build (`en`, `de`, `fr`, …) loaded lazily.

### 3.2 Mobile & Desktop (KMP / Compose)

- **moko-resources** for strings + plurals + format strings.
- Shared `commonMain/resources/MR/base/strings.xml` (English source).
- Locale overrides in `MR/<locale>/strings.xml`.
- Runtime locale switch without app restart.

### 3.3 Backend (Go)

- Server-generated user-visible strings are rare (we prefer to return structured errors and let the client render).
- When unavoidable (email templates, webhooks with body text, PDFs), use `golang.org/x/text/message` with message catalogs.
- Templates in `templates/emails/<locale>/*.tmpl`.

### 3.4 Notification Content

- Email subjects + bodies per locale.
- Fallback to English if target locale missing.
- Respect user's preferred language (`users.locale` column).

## 4. Translation Platform

We use **Crowdin** (or **Lokalise** / **Transifex** — equivalent flow).

- CI action on every merge to `main`:
  1. Extract new keys from source.
  2. Push to translation platform.
  3. Open PR to commit any newly completed translations back to the repo.
- **Glossary** per locale committed to the platform; enforces consistent terminology for domain words (e.g. "pull request", "conflict case").
- **Context screenshots** attached automatically where possible (web storybook renders → images pushed alongside keys).
- Translation progress dashboard in Grafana.

---

## 5. Message Key Conventions

- Key path: `<area>.<screen>.<element>.<role>` — e.g. `repo.settings.branch_protection.title`.
- Plural/ICU: one key per *unit* of meaning; let ICU handle plural forms.
  ```json
  "repo.counts.open_issues": "{count, plural, =0 {No open issues} one {# open issue} other {# open issues}}"
  ```
- Never concatenate: **every user sentence is one key**.
- Avoid gendered English phrasing that other languages must split.

---

## 6. Non-String Localisation

### 6.1 Dates & Times

- Use `Intl.DateTimeFormat` (web/KMP) / `java.time.format.DateTimeFormatter.ofLocalizedDateTime` / `DateFormat` (iOS).
- Store all datetimes as **UTC ISO 8601**; format on display.
- Allow user to set their preferred timezone explicitly.
- Relative times ("3 minutes ago") use locale-aware formatters (`Intl.RelativeTimeFormat`).

### 6.2 Numbers & Currencies

- `Intl.NumberFormat` with currency when applicable.
- Never assume locale uses `,` or `.` as decimal separator.
- Display prices in the user's plan currency (default USD) with correct symbol position and grouping.

### 6.3 Names & Addresses

- Name fields: single `display_name` (allow user self-entered). No assumption of first/last ordering.
- Addresses: always free-form multi-line; let the user paste.

### 6.4 Text Direction

- Set `dir` attribute per locale.
- CSS logical properties (`margin-inline-start`, `padding-inline-end`) instead of physical.
- Mirror icons that imply direction (arrows, progress indicators).

---

## 7. Pluralisation & Gender

- **Plural**: use ICU `plural`. Source English covers `one` and `other`; some locales need `zero`, `two`, `few`, `many`, `other` — translators supply.
- **Gender**: use ICU `select` on gendered subjects. Avoid where feasible by rewriting.

Example:

```json
"conflict.assigned": "{gender, select, female {She was assigned a conflict} male {He was assigned a conflict} other {{name} was assigned a conflict}}"
```

We prefer "other" everywhere — inclusive by default.

---

## 8. Accessibility Intersection

- Screen-reader pronunciation: set proper `lang` attributes on mixed-language content so ATs switch voice.
- RTL mirroring preserves reading order for ATs.
- Language-specific icon/emoji avoidance where ambiguous.

---

## 9. Testing

- **Unit**: snapshot tests per locale for key screens; CI blocks on missing keys.
- **Visual regression**: Chromatic renders each locale variant on critical screens.
- **Pseudo-localisation**: `xx-PL` pseudo-locale doubles string length and wraps in brackets; exposes hardcoded strings and clipping issues. CI visual-diffs a pseudo build.
- **RTL visual regression**: `ar-XA` pseudo-RTL build.
- **Linting**: `i18n-lint` rejects bare JSX text nodes; ESLint rule `i18n/no-raw-strings`.

---

## 10. Performance

- Web: per-locale JS bundle; client downloads only its locale (~20 KB typical).
- Mobile/Desktop: all strings shipped in the binary for offline. Desktop with many locales gets ~2 MB of strings; acceptable.

---

## 11. Process

- New feature lands with English strings + key IDs.
- Crowdin picks up the delta within 24 h.
- Volunteer translators + staff translators fill in.
- PR opened when target coverage reached (≥ 95 % for release-blocking).
- Translation reviewers gate merges per locale.
- Product decides whether to release with partial translation (falls back to English gracefully).

---

## 12. Translation Contribution (Community)

- Crowdin public project.
- Credit per locale on the docs site.
- Moderator roles for trusted contributors.
- Code of conduct; disputes escalate to product.

---

## 13. Handling Ambiguity

- Reviewers add context notes where English is ambiguous (e.g. "button vs. header").
- Screenshots attached to keys automatically.
- Glossary is authoritative; deviations tracked per locale.

---

## 14. Emoji / Icons

- Where used as status markers, always pair with text for a11y + translatability.
- Avoid flags to represent languages (political sensitivity).

---

## 15. Date / Number Gotchas

- Week start: locale-specific (Sunday vs. Monday). Respect user's calendar settings.
- Numeric year 2-digit display: prefer 4-digit (`yyyy`), especially near century boundaries.
- Ordinals ("1st", "2nd"): locale-specific; use formatter.
- Currency ISO codes: always available in API responses; UI formats per locale.

---

## 16. Content Management

- Marketing site and docs use the same translation platform.
- Docs: section-level translation; if behind, English shown with a banner "This section hasn't been translated yet."
- Blog posts: translated on demand; not every post translated.

---

## 17. Governance

- **i18n Council**: product, frontend lead, mobile lead, legal (for region-specific terms), one translator-community rep.
- Meets quarterly.
- Decides locale additions, removals, quality thresholds.
